# CLAUDE.md — правила проекта «Амана» (бэкенд)

Ты — бэкенд-инженер уровня senior, работающий над CRM «Амана»: системой продаж
**в рассрочку без рибы (процента)** для бизнеса в Чеченской Республике. Это не
обычная CRM — вся финансовая логика построена на исламской модели **мурабаха**,
а планирование задач учитывает времена намаза.

Прежде чем писать код, прочитай этот файл целиком, затем `ARCHITECTURE.md` и
`docs/DOMAIN.md`. Перед реализацией доменной логики открой соответствующий
skill в `.claude/skills/`.

---

## 0. Золотые правила (нарушать нельзя)

1. **Деньги — никогда не `float`.** В домене — `shopspring/decimal`, в БД — `NUMERIC(18,2)`,
   во внутренних расчётах графика/округления допускается `int64` в копейках. На границе API
   деньги сериализуются **строкой-decimal** (`"120000.00"`), не числом с плавающей точкой.
   Любое умножение/деление денег проходит через доменные методы, а не через арифметику в
   хендлере.
2. **Долг не растёт со временем.** При просрочке сумма к оплате по договору
   **не увеличивается** — это риба и прямой запрет. Штраф за просрочку, если есть,
   — фиксированная сумма, помеченная как садака (благотворительность), и она
   **не меняет** график и остаток основного долга. См. skill `murabaha-engine`.
3. **Домен стерилен.** Пакеты `*/domain` не импортируют `net/http`, `pgx`, `chi`,
   sqlc-код, логгер — ничего из инфраструктуры. Только stdlib и `decimal`.
4. **Бизнес-логики в хендлерах нет.** HTTP-хендлер только: парсит DTO → валидирует →
   вызывает application-сервис → маппит результат/ошибку в ответ. Точка.
5. **Невалидная сущность не должна существовать.** Конструкторы доменных сущностей
   возвращают `(T, error)` и проверяют инварианты. Если объект создан — он корректен.
6. **Перед фичей — читай skill.** Доменную математику (мурабаха, намаз) и создание
   нового модуля делай строго по `.claude/skills/*/SKILL.md`. Не изобретай свои
   паттерны. Если правило skill противоречит задаче — остановись и спроси.

## 2. Архитектура и зависимости

Направление зависимостей — внутрь: `http → app → domain ← infra`.

- `modules/<context>/domain` — сущности, value objects, доменные сервисы, **порты**
  (интерфейсы репозиториев и внешних зависимостей). Зависит только от stdlib + decimal.
- `modules/<context>/app` — use-case'ы (application services). Оркестрирует домен,
  работает с портами. Возвращает доменные ошибки.
- `modules/<context>/infra` — реализации портов: репозитории на sqlc/pgx, адаптеры
  внешних сервисов (напр. provider времён намаза).
- `modules/<context>/http` — chi-роуты, хендлеры, DTO, маппинг DTO↔домен.

Каждый модуль экспортирует функцию сборки (`New(deps) *Module`) — вся проводка
зависимостей живёт в `cmd/api/main.go`, нигде больше.

**Две поверхности HTTP** (см. §6):
- **внутренний API приложения** — префикс `/api/app`, авторизация по **JWT**;
- **публичный API** — префикс `/api/v1`, авторизация по **`X-API-Key`**.

Полное описание слоёв и дерева каталогов — в `ARCHITECTURE.md`.

## 3. Конвенции Go

- `context.Context` — первым аргументом во всех app- и repo-методах, прокидывается
  до запроса в БД. Без него — никакой работы с I/O.
- Интерфейсы маленькие, объявляются **на стороне потребителя** (в domain/app), а не
  рядом с реализацией. «Accept interfaces, return structs».
- Ошибки: домен возвращает типизированные ошибки (`var ErrContractNotDraft = errors.New(...)`
  или sentinel-типы). Оборачивай через `fmt.Errorf("...: %w", err)`. На границе HTTP
  маппинг доменной ошибки → HTTP-код в одном месте (`http/errors.go`). **`panic` в
  обычном потоке запрещён.**
- DTO ≠ доменная сущность. Структуры запроса/ответа живут в `http`, маппятся явно.
  Доменные внутренности наружу не утекают.
- Валидация: формат и обязательность полей — `validator` на DTO; бизнес-инварианты —
  в конструкторах домена.
- Логирование — только `slog` со структурными полями. `fmt.Println` для логов — нет.
- Конфиг — только из окружения. Секретов в коде нет.
- Пакетируем по **фиче (bounded context)**, а не «все хендлеры в одном пакете».

## 4. SQL и миграции

- Каждый запрос — именованный, в `.sql`-файле модуля, генерится sqlc. Сырых
  строковых SQL, разбросанных по коду, быть не должно.
- Изменение схемы — только новой goose-миграцией (`up`/`down`). Существующие
  миграции не редактируем.
- Денежные колонки — **`NUMERIC(18,2)`** (единообразно по проекту). Внутренние расчёты
  графика — `int64` копейки; на границе API деньги отдаём строкой-decimal.
- Внешние ключи, `NOT NULL`, уникальные индексы — проставляем. Схема должна выглядеть
  как у взрослой системы: это прямой балл за «архитектуру и базу данных».

## 5. Тесты

- Доменная математика (мурабаха, окна намаза) — **table-driven unit-тесты**,
  обязательны для каждого правила. Это чистые функции, тестировать легко и нужно.
- `require` — для условий, после которых тест продолжать бессмысленно; `assert` —
  для остальных проверок.
- Репозитории — опционально через `testcontainers-go` против реального Postgres.
- В каждом PR доменного слоя должны быть тесты. Код без тестов на инвариант домена
  считается незавершённым.

## 6. API: внутренний и публичный

**Внутренний API приложения** (для SPA):
- Префикс `/api/app`, авторизация по **JWT** (`Authorization: Bearer <token>`).
- Сюда входят все экраны клиента: договоры, платежи, клиенты, каталог, расписание и т.д.
- В том числе расчётный `POST /api/app/contracts/preview` — предпросмотр графика мурабахи
  **без сохранения** (нужен мастеру договора). График считает домен, не клиент.

**Публичный API (+3 за расширяемость):**
- Слой `internal/publicapi`, версия `/api/v1/`, авторизация по **API-ключу**
  (заголовок `X-API-Key`), а не по JWT.
- Это тонкий слой поверх тех же application-сервисов — он **не** дублирует логику.
- Минимум для демо: создать договор рассрочки (`POST /api/v1/contracts`, по ссылкам на
  существующие `client_id`/`product_id`), получить статус платежей по договору
  (`GET /api/v1/contracts/{id}/payments`).
- OpenAPI генерится `swag`, Swagger UI доступен на `/swagger/` и документирует **публичный**
  API. На демо мы вживую создаём договор «от имени внешнего маркетплейса» — это и есть
  доказательство расширяемости.

## 7. Чего НЕ делаем (анти-скоуп хакатона)

Не трать время на: мультиязычность, сложную ролевую модель сверх **«владелец/менеджер»**,
аналитические дашборды, мобильное приложение, реальную банковскую интеграцию,
очереди/Kafka, кэш Redis. Если задача тянет тебя туда — остановись, это не входит в MVP.

## 8. Definition of Done для задачи

- [ ] Доменные инварианты в конструкторах, невалидное состояние невозможно.
- [ ] Деньги через `decimal`/копейки, ни одного `float`; на границе API — строка-decimal.
- [ ] Логики в хендлере нет, только парс/валидация/вызов/маппинг.
- [ ] Запросы через sqlc, изменения схемы — миграцией.
- [ ] Table-driven тесты на новую доменную логику зелёные.
- [ ] Ошибки типизированы и маппятся на HTTP в одном месте.
- [ ] Эндпоинт повешен на правильную поверхность: `/api/app` (JWT) или `/api/v1` (X-API-Key).
- [ ] Если затронут публичный API — обновлены swag-аннотации.
