# Skill: планировщик с учётом намаза

Задачи (звонок, доставка, контакт по платежу) нельзя назначать на время намаза и на
пятничный джума-намаз. Желаемое время сдвигается на ближайший свободный слот.

## Порт (домен зависит от интерфейса, не от библиотеки)

```go
type PrayerTimes struct {
    Fajr, Dhuhr, Asr, Maghrib, Isha time.Time
}

type PrayerTimesProvider interface {
    For(ctx context.Context, date time.Time, loc Location) (PrayerTimes, error)
}
```

- Доменный сервис планирования принимает `PrayerTimesProvider`. Реализация —
  в `infra` поверх `hablullah/go-prayer`. **Домен про библиотеку не знает.**

## Параметры расчёта (для Грозного)

- Координаты Грозного: `lat = 43.3178`, `lon = 45.6949`, таймзона `Europe/Moscow` (MSK, UTC+3).
- Метод расчёта углов — подходящий для региона (напр. Muslim World League), вынести в конфиг.
- **Мазхаб для расчёта Аср — шафиитский** (стандартный shadow factor = 1). Чеченцы
  преимущественно шафииты; это влияет на время Аср. Параметр конфигурируемый, дефолт — шафиитский.

## Построение заблокированных окон

Для даты `D` и времён намаза строим интервалы блокировки:

- Для каждой из 5 молитв `p`: окно `[p − BufferBefore, p + BufferAfter]`
  (дефолт `BufferBefore = 0`, `BufferAfter = 20 мин`; конфигурируемо).
- Если `D` — **пятница**: добавить окно джума `[JummahStart, JummahEnd]`
  (дефолт 12:30–14:00 по местному времени; конфигурируемо). В это время бизнес обычно
  приостанавливается, окно шире обычного.

Все границы — в локальном времени Грозного.

## Алгоритм поиска ближайшего свободного слота

Вход: желаемое начало `T`, длительность `dur` (для доставки; для звонка `dur` мал/0),
политика (буферы, окно джума).

```
1. windows := построить заблокированные окна для даты T
2. slot := [T, T + dur]
3. пока slot пересекается с каким-либо окном w из windows:
       T := w.End                      // сдвигаем к концу мешающего окна
       slot := [T, T + dur]
       (повтор — сдвиг мог попасть в следующее окно)
4. вернуть { Time: T, WasShifted: (T != исходного), Reason: какое окно мешало }
```

- Сдвиг — **вперёд** (ближайшее доступное после желаемого). Политику «можно раньше»
  не вводим без явного требования.
- Если сдвиг выходит за конец рабочего дня — допускается перенос на следующий день
  (если это правило включено); иначе вернуть слот как есть и пометить флагом.
- Алгоритм детерминированный и завершается (число окон конечно, каждый шаг двигает `T`
  строго вперёд).

## Сущность результата

```go
type ScheduledTime struct {
    Time       time.Time
    WasShifted bool
    Reason     string // например: "сдвинуто из-за Магриб" / "сдвинуто из-за джума"
}
```

`WasShifted` и `Reason` показываем в UI — это часть «вау» демо: система сама объяснила,
почему перенесла действие.

## Обязательные тесты (table-driven, с фиктивным PrayerTimesProvider)

Используем стаб провайдера с фиксированными временами намаза, чтобы тесты были
детерминированы (не зависели от реального расчёта):

1. **Время вне окон** → не сдвигается, `WasShifted == false`.
2. **Время внутри окна молитвы** → сдвигается на `prayer + BufferAfter`,
   `WasShifted == true`, корректный `Reason`.
3. **Доставка с длительностью задевает окно** → старт сдвигается так, чтобы весь
   интервал `[T, T+dur]` был вне окна.
4. **Каскадный сдвиг** → после сдвига попадаем в следующее окно, проверяем второй сдвиг.
5. **Пятница, попадание в окно джума** → сдвиг за `JummahEnd`, `Reason` про джума.
6. **Не-пятница** → окно джума не применяется.
7. **Граничные касания** (конец одного окна = желаемое время) → нет ложного сдвига.
