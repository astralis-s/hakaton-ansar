# ARCHITECTURE.md — «Амана», бэкенд

## Тип системы: модульный монолит

Один деплоируемый бинарь, внутри — изолированные bounded contexts с границами
через интерфейсы. Это сознательный выбор против микросервисов.

**Почему не микросервисы (для хакатона и не только):**
- Микросервисы дают распределённую сложность (сеть, согласованность, N деплоев)
  без выгод, пока нагрузки и команды маленькие.
- Модульный монолит на чистой архитектуре масштабируется по тому же принципу:
  у каждого контекста свои интерфейсы и своя схема данных, поэтому любой модуль
  **вытаскивается в отдельный сервис без переписывания домена** — меняется только
  реализация портов (вместо вызова в памяти — gRPC/HTTP-клиент).
- На демо это сильнее: «вот границы сервисов, вот один бинарь, который ставится одной
  командой» закрывает сразу и «масштабируемость», и «простоту развёртывания».

## Слои (чистая / гексагональная архитектура)

Правило зависимостей — строго внутрь. Внешние слои знают про внутренние, не наоборот.

```
        HTTP (chi handlers, DTO)
                 │  вызывает
                 ▼
     Application (use-cases, оркестрация)
                 │  использует порты
                 ▼
        Domain (сущности, VO, доменные
        сервисы, ПОРТЫ-интерфейсы)
                 ▲  реализует порты
                 │
   Infrastructure (sqlc/pgx репозитории,
        адаптер времён намаза, ...)
```

- **Domain** — сердце. Бизнес-правила, инварианты, value objects, доменные сервисы.
  Объявляет **порты** (интерфейсы): `ContractRepository`, `PaymentRepository`,
  `PrayerTimesProvider` и т.п. Импортирует только stdlib и `shopspring/decimal`.
- **Application** — сценарии («оформить договор», «принять платёж», «предпросмотр графика»,
  «запланировать напоминание»). Тонкий слой: достаёт агрегаты через порты, вызывает доменные
  методы, сохраняет. Транзакции — здесь.
- **Infrastructure** — реализации портов. Репозитории на sqlc+pgx, адаптеры внешних
  библиотек. Зависит от domain (реализует его интерфейсы), не наоборот.
- **HTTP** — транспорт. Роуты, хендлеры, DTO, валидация формата, маппинг ошибок.

**Две поверхности HTTP:** внутренний API приложения (`/api/app`, JWT) и публичный API
(`/api/v1`, `X-API-Key`). Обе — тонкие транспортные слои над одними и теми же
application-сервисами; логику не дублируют.

## Поток одного запроса (пример: «принять платёж»)

1. `POST /api/app/contracts/{id}/payments` → chi-роут модуля `financing` (внутренний API, JWT).
2. Хендлер парсит `RegisterPaymentRequest` (DTO), валидирует формат через `validator`.
3. Вызывает `app.RegisterPayment(ctx, contractID, amount)`.
4. App открывает транзакцию, грузит агрегат `Contract` через `ContractRepository`.
5. Доменный метод `contract.RegisterPayment(amount)` проверяет инварианты
   (`0 < amount ≤ Outstanding`, статус активен), уменьшает `Outstanding`, пересчитывает
   производные статусы долей (оплачен/частично оплачен/предстоит), при нуле → статус `Completed`.
6. App сохраняет агрегат, коммитит.
7. Хендлер маппит результат в `PaymentResponse` (деньги — строкой-decimal) либо доменную
   ошибку в HTTP-статус.

## Персистентность

- `sqlc` генерит типобезопасные методы из `.sql`. Репозиторий из `infra` реализует
  доменный порт, внутри вызывает sqlc-методы и маппит строки БД ↔ доменные сущности.
- Денежные колонки — `NUMERIC(18,2)`; в домене — `Money` (decimal), во внутренних расчётах
  графика — `int64` копейки.
- Транзакции прокидываются через `pgx.Tx`; helper в `platform/database` даёт
  `WithinTx(ctx, fn)`.
- Миграции `goose` эмбедятся в бинарь и прогоняются на старте (флаг `MIGRATE_ON_BOOT`).

## Сборка зависимостей (DI)

Без DI-фреймворков. Вся проводка — руками в `cmd/api/main.go`: создаём пул pgx,
логгер, конфиг; затем `financing.New(pool, logger)`, `scheduling.New(pool, prayerProvider)`
и т.д.; монтируем внутренние роуты под `/api/app`, публичные — под `/api/v1`; собираем общий
`chi.Router`; поднимаем `httpserver`.

## Деплой (критерий «простота развёртывания»)

- Multi-stage `Dockerfile`: сборка статического бинаря → финальный образ на distroless.
  Статика фронта (`web/dist`) эмбедится в бинарь либо раздаётся рядом; SPA-fallback на
  `index.html`.
- `docker-compose.yml` поднимает `postgres` + `app`. Запуск проекта = **одна команда**
  `docker compose up`. Миграции прогоняются автоматически на старте.
- `Makefile` для разработчика: `make run`, `make test`, `make migrate`, `make sqlc`,
  `make swag`, `make lint`.

## Как вытащить модуль в микросервис (история масштабирования для жюри)

Домен и app уже зависят только от портов. Чтобы вынести `financing` в сервис:
заменяем in-memory вызов его app-сервиса на gRPC-клиент за тем же интерфейсом,
поднимаем модуль отдельным `cmd/financing/main.go`. Домен не трогаем. Это и есть
смысл чистой архитектуры — мы платим за неё ровно тогда, когда выгода появилась.
