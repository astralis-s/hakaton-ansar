# DOMAIN.md — доменная модель «Амана»

Документ фиксирует **единый язык** (ubiquitous language) и тактические объекты DDD.
Бизнес-термины — по-русски, идентификаторы в коде — по-английски.

## Единый язык

| Термин (бизнес)        | Код (English)        | Значение |
|------------------------|----------------------|----------|
| Договор рассрочки      | `Contract`           | Соглашение мурабаха: продажа товара с фиксированной наценкой в рассрочку |
| Закупочная цена        | `CostPrice`          | За сколько продавец приобрёл товар |
| Наценка                | `Markup`             | Фиксированная надбавка продавца. Задаётся **суммой** (не ставкой) и больше не меняется |
| Цена продажи           | `SalePrice`          | `CostPrice + Markup`. Полная сумма обязательства клиента, фиксируется при создании |
| Первоначальный взнос   | `DownPayment`        | Что клиент платит сразу (может быть 0) |
| Сумма к рассрочке      | `FinancedAmount`     | `SalePrice − DownPayment` |
| График платежей        | `PaymentSchedule`    | Набор плановых платежей с датами |
| Плановый платёж        | `Installment`        | Одна доля графика: сумма + срок + статус |
| Фактический платёж     | `Payment`            | Зарегистрированный приход денег по договору |
| Остаток долга          | `Outstanding`        | Сколько ещё должен клиент. Не растёт со временем. **Источник правды по прогрессу** |
| Садака за просрочку    | `LatePenaltyCharity` | Фиксированный сбор при просрочке, идёт на благотворительность, не продавцу |
| Халяль-статус          | `HalalStatus`        | `Halal` / `Haram` / `Doubtful (mushbooh)` |
| Окно намаза            | `PrayerWindow`       | Интервал, в который нельзя планировать действия |
| Напоминание / задача   | `Reminder`           | Запланированное действие (звонок, доставка, контакт по платежу) |

## Bounded contexts

- **financing** — ★ ядро. Договоры мурабахи, графики, платежи. Здесь живёт вся
  финансовая корректность. Подробности — в skill `murabaha-engine`.
- **scheduling** — планирование действий с обходом времён намаза. Подробности —
  в skill `namaz-scheduler`.
- **catalog** — товары, закупочная цена, халяль-статус.
- **crm** — клиенты (ФИО, контакты, документ для договора).
- **iam** — пользователи, аутентификация (JWT), API-ключи для публичного API.

## Контекст financing (ядро)

### Агрегат: Contract (корень агрегата)

Внутри агрегата — `PaymentSchedule` (список `Installment`) и список `Payment`.
Изменять график и остаток можно **только** через методы корня — это защищает инварианты.

Поля (по смыслу):
- `ID`, `ClientID`, `ProductID`
- `CostPrice : Money`, `Markup : Money`, `SalePrice : Money` (= Cost + Markup)
- `DownPayment : Money`
- `Schedule : PaymentSchedule`
- `Payments : []Payment`
- `Outstanding : Money`
- `Status : ContractStatus`
- `CreatedAt`, `Cadence` (периодичность платежей, напр. месяц)

### Value Objects

- **`Money`** — сумма + валюта. Внутри — `decimal.Decimal`; во внутренних расчётах графика и
  округления нормализуется в `int64` копейки (детерминированно). Только через её методы:
  `Add`, `Sub`, `IsNegative`, `Equals`, деление с остатком. Прямой арифметики над деньгами
  вне `Money` быть не должно. **Хранение** — `NUMERIC(18,2)`; **на границе API** — строка-decimal
  (`"120000.00"`). Ни одного `float`.
- **`Markup`** — обёртка над `Money` с гарантией `>= 0`. Может конструироваться из
  процента от `CostPrice`, но после конструирования это **фиксированная сумма**.
- **`ContractStatus`** — перечисление со state machine (ниже).
- **`InstallmentStatus`** — перечисление `Pending` / `PartiallyPaid` / `Paid` / `Overdue`
  (см. «Статусы доли»).

### Инварианты (проверяются в конструкторе/методах)

1. `SalePrice == CostPrice + Markup` — всегда.
2. `0 <= DownPayment < SalePrice`.
3. `FinancedAmount >= Installments` — каждая доля получает **минимум 1 копейку**; нулевых
   долей быть не может (иначе конструктор возвращает ошибку).
4. `сумма всех Installment == FinancedAmount` (с точностью до копейки).
5. `DownPayment + сумма всех Installment == SalePrice`.
6. `Outstanding` стартует как `SalePrice − DownPayment`, монотонно не возрастает,
   никогда не отрицателен.
7. **Сумма обязательства фиксируется при создании и не зависит от времени.** Просрочка
   не меняет `SalePrice`, `Outstanding` или график.
8. `0 < Payment.Amount <= Outstanding` на момент регистрации.

### Статусы доли (Installment)

`Outstanding` — единственный источник правды по прогрессу; статусы долей **производны** от
накопленной фактической оплаты, а не хранятся независимо:

- **`Paid`** — суммарная оплата ≥ верхней (накопительной) границы этой доли в графике.
- **`PartiallyPaid`** — суммарная оплата попадает внутрь этой доли (ровно одна такая доля).
- **`Pending`** — доля ещё не покрыта и срок не наступил.
- **`Overdue`** — доля не покрыта и срок прошёл (вспомогательный флаг `HasOverdueInstallment`
  на договоре). Это **не** про рост долга.

### Предпросмотр графика

`PreviewContract` — **чистый расчёт** графика, цены продажи и данных сравнения по тем же
правилам мурабахи, **без создания агрегата и без записи в БД**. Используется мастером
договора (эндпоинт `POST /api/app/contracts/preview`), чтобы показать график до оформления.
Клиент график не считает.

### Жизненный цикл (state machine)

```
Draft ──activate──► Active ──(Outstanding == 0)──► Completed
  │                   │
  │                   └──settleEarly──► Completed   (оплата остатка целиком, без штрафа)
  └──cancel──► Cancelled            Active ──cancel──► Cancelled
```

- Просрочка **не** переводит договор в отдельный «штрафной» статус с ростом долга.
  Допустим вспомогательный флаг `HasOverdueInstallment`, но это не про увеличение суммы.
- Досрочное погашение разрешено всегда и без штрафа (запрет на штраф за досрочку).
- **Границы UI (MVP).** State machine поддержана в домене полностью. В интерфейсе: мастер
  договора выполняет **create+activate атомарно** (договор сразу `Active`); `Draft` —
  внутреннее/для публичного API состояние, отдельным экраном **не** показывается; переход в
  `Cancelled` доступен простым действием **«Отменить договор» (только владелец)** на карточке.

### Поведение при просрочке (анти-риба)

- Сумма долга не меняется.
- Если бизнес-настройка включает сбор за просрочку — это **фиксированная** величина
  (`LatePenaltyCharity`), помеченная как благотворительная: учитывается в отдельном
  «реестре садаки», **не** в выручке продавца и **не** в `Outstanding`. Начисление садаки —
  действие **владельца** (см. `docs/SITE_STRUCTURE.md`).
- Подробные формулы, округление и тест-кейсы — в skill `murabaha-engine`.

## Контекст scheduling

### Сущности

- **`Reminder`** — тип (`Call` / `Delivery` / `PaymentFollowUp`), желаемое время,
  длительность (для доставки), итоговое время после обхода намаза, флаг `WasShifted`,
  статус `Scheduled` / `Completed` / `Cancelled`. Состояние **`Overdue` не хранится**:
  это производное представление для UI, когда запланированная задача ещё не закрыта,
  а её `scheduled_at` уже прошло.
- **`PrayerWindow`** — интервал блокировки `[start, end]` вокруг молитвы.

### Порт

- **`PrayerTimesProvider`** — `PrayerTimes(ctx, date, location) -> {Fajr, Dhuhr, Asr, Maghrib, Isha}`.
  Доменный сервис планирования зависит от этого интерфейса, **не** от библиотеки.
  Реализация в `infra` (через `go-prayer`), параметры — шафиитский мазхаб, координаты Грозного.

### Правило

Желаемое время сдвигается так, чтобы действие не попадало в окно намаза и в окно
пятничного джума-намаза. Алгоритм поиска ближайшего свободного слота и тест-кейсы —
в skill `namaz-scheduler`.

## Контекст catalog

- **`Product`** — `Name`, `CostPrice : Money`, `HalalStatus`, `Category`.
- `HalalStatus` обязателен. Договор на товар со статусом `Haram` создавать нельзя
  (доменная проверка на этапе создания `Contract`).

## Контекст crm

- **`Client`** — `FullName`, контакты, идентификационный документ (нужен для договора).
  Намеренно простой: глубокая воронка продаж в скоуп не входит.

## Контекст iam

- **`User`** — роль **`владелец` / `менеджер`** (`RoleOwner` / `RoleManager`), вход во
  внутренний API приложения (`/api/app`) по JWT, пароль через bcrypt.
- **`ApiKey`** — ключ для **публичного** API (`/api/v1`, заголовок `X-API-Key`), привязан к
  организации, отзываемый. Не путать с пользовательскими JWT.
- Публичный `POST /api/v1/contracts` принимает **ссылки на существующие** `client_id` и
  `product_id` (клиент и товар создаются заранее — через приложение или сидированием для демо).
