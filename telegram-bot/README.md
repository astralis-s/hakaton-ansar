# Telegram-бот «Амана» — мост заказчик ↔ менеджер

Автономный сервис: заказчики пишут **Telegram-боту**, их сообщения попадают в
**веб-инбокс менеджера**; ответы менеджера с сайта доставляются обратно в Telegram.

При первом запуске бот проводит **регистрацию**: спрашивает ФИО и контактный
телефон. Только после этого заказчик может писать менеджеру. В списке чатов на
сайте отображается именно **ФИО из регистрации**, а не никнейм Telegram.

> ⚠️ Папка полностью изолирована от основного бэкенда CRM: **свой `go.mod`,
> только стандартная библиотека Go, ноль внешних зависимостей**, никакого общего
> кода. Её можно собирать, запускать и удалять, не задевая остальной проект.

## Как это работает

```
                          ┌──────────────────────────────┐
   Telegram (заказчик)    │          amana-bot           │     Менеджер (браузер)
        │                 │                              │            │
        │  /start, текст  │  long polling getUpdates     │            │
        ├────────────────▶│  ── FSM регистрации ───────▶ │            │
        │                 │     (ФИО → телефон)          │            │
        │   сообщение     │                              │  GET /api/chats (опрос)
        ├────────────────▶│  store: сообщение клиента ──▶│───────────▶│  список чатов (ФИО, телефон)
        │                 │                              │            │
        │   ответ         │  sendMessage  ◀── SendFromManager ◀───────┤  POST .../messages
        │◀────────────────┤                              │            │  (ответ менеджера)
        │                 └──────────────────────────────┘            │
```

- **Long polling** Telegram Bot API (`getUpdates`) — публичный URL/вебхук не нужен.
- **Хранилище** — потокобезопасный JSON-файл (`DATA_FILE`). Пользователи и вся
  переписка переживают перезапуск. Внешняя БД не требуется.
- **Веб-инбокс** — встроенная одностраничная статика (`/`) + JSON API (`/api/...`),
  опрашивает сервер раз в пару секунд (без вебсокетов).

## Структура

```
telegram-bot/
├── cmd/bot/main.go              # точка входа: конфиг → store → telegram → bridge → http
├── internal/
│   ├── config/                  # настройки из окружения (+ мини-загрузчик .env)
│   ├── store/                   # JSON-хранилище: пользователи, сообщения, инбокс
│   ├── telegram/                # мини-клиент Bot API (getUpdates/sendMessage) + поллер
│   ├── app/                     # мост: FSM регистрации, релей сообщений (+ тесты)
│   └── webui/                   # HTTP API менеджера + встроенная статика (+ тесты)
└── go.mod                       # отдельный модуль, только stdlib
```

## Запуск

1. Создайте бота у [@BotFather](https://t.me/BotFather) и получите токен.
2. Скопируйте `.env.example` в `.env` и впишите `BOT_TOKEN` (и пароль менеджера):

   ```bash
   cp .env.example .env
   # отредактируйте BOT_TOKEN и MANAGER_PASSWORD
   ```

3. Запустите:

   ```bash
   make run          # или: go run ./cmd/bot
   ```

4. Менеджерский инбокс откроется на `http://localhost:8090` (см. `HTTP_ADDR`).
   Заказчик пишет боту в Telegram `/start` — и дальше по сценарию регистрации.

### Переменные окружения

| Переменная         | По умолчанию          | Описание                                            |
|--------------------|-----------------------|-----------------------------------------------------|
| `BOT_TOKEN`        | —                     | Токен бота от @BotFather (**обязательно**).          |
| `HTTP_ADDR`        | `:8090`               | Адрес веб-инбокса менеджера.                          |
| `MANAGER_PASSWORD` | —                     | Пароль входа в инбокс. Пусто → вход без пароля.       |
| `DATA_FILE`        | `data/bot-data.json`  | Файл JSON-хранилища.                                  |
| `POLL_TIMEOUT_SEC` | `30`                  | Таймаут long polling getUpdates.                      |

## Сценарий регистрации (FSM)

1. `/start` (или первое сообщение) → бот просит **ФИО** одной строкой.
2. Принимает ФИО (минимум фамилия + имя) → просит **телефон**
   (кнопка «отправить мой номер» или ввод вручную, нормализуется к `+7…`).
3. После телефона — статус «зарегистрирован», клавиатура убирается. Теперь любое
   сообщение уходит менеджеру в инбокс, а ответы менеджера приходят сюда же.

## Тесты

Покрыты: нормализация телефона и ФИО, полный путь FSM регистрации, релей сообщений

## Заметки по дизайну

- Ответ менеджера пишется в историю **только после успешной доставки** в Telegram —
  в инбоксе не появляется «отправленных», которые на самом деле не дошли.
- В инбоксе показываются только **завершившие регистрацию** пользователи (с ФИО и
  телефоном) — недо-зарегистрированные не мусорят список.
- Хранилище спрятано за методами `store.Store`; при желании JSON-файл легко заменить
  на SQLite/Postgres, не трогая остальной код.
- Если в будущем понадобится завести этих заказчиков как клиентов основной CRM —
  бот общается с системой только через своё хранилище, так что интеграцию можно
  добавить отдельным адаптером, не меняя текущую логику.
