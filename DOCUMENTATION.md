# Амана (Amana) — проектная документация

> **CRM для честной рассрочки без рибы.**
> Система для бизнеса Чеченской Республики, который продаёт товары в рассрочку по
> исламской модели **мурабаха**: фиксированная цена без процента, садака вместо
> штрафов и планирование задач с учётом времён намаза.

**Имя:** Амана (от араб. *аманат* — доверенное на хранение, ответственность за честность).
**Тип продукта:** веб-CRM (SaaS, мультиорганизационная).
**Кейс хакатона:** CRM-система (тип задан, продукт — наш).

---

## 1. Кратко (elevator pitch)

Обычная CRM зашивает в логику процентную ставку — в регионе, где **риба (процент)
запрещена**, такая система нерабочая. «Амана» переписывает саму финансовую модель:
цена товара фиксируется один раз как «себестоимость + наценка», долг **не растёт со
временем**, а вместо пени за просрочку — фиксированная **садака на благотворительность**.
Сверху — планировщик, который не назначает звонки и доставки на время намаза, и
**собственный публичный API** для интеграции внешних сервисов.

## 2. Контекст и проблема

- В Чеченской Республике мебель, технику и авто массово продают **в рассрочку**, но
  по нормам ислама недопустим процент (риба). Продавцы ведут такие сделки в таблицах
  и переписке — без контроля графиков, остатков и просрочек.
- Существующие CRM спроектированы вокруг кредита с процентной ставкой и капитализацией
  пени. Это не косметическое несоответствие — **другая бизнес-логика**.
- Регион на подъёме: по данным правительства ЧР (2025), за год привлечено более 171 млрд ₽
  инвестиций (рост ≈5% г/г), активно растут туризм и розница. Спрос на цивилизованные
  инструменты для местного бизнеса — реальный.

**Вывод:** узкая ниша с понятной болью и логикой, которой нет ни у одной массовой CRM.

## 3. Целевые пользователи

| Роль | Кто это | Что делает в системе |
|------|---------|----------------------|
| Владелец | Хозяин магазина (мебель/техника/авто) | Видит общую картину, настраивает правила бизнеса |
| Менеджер | Продавец-консультант | Заводит клиентов, оформляет договоры, принимает платежи, ведёт напоминания |
| Внешний сервис | Маркетплейс / сайт партнёра | Через публичный API создаёт договоры и читает статусы (интеграция, +3) |

## 4. Ценностное предложение (почему это выигрывает)

Три фишки, на которых держится продукт, — и каждая закрывает сразу несколько критериев:

1. **Движок мурабахи без рибы** — фиксированная цена, долг не растёт, садака вместо
   пени, досрочное погашение без штрафа. Это ядро и главное «ноу-хау».
2. **Планировщик с учётом намаза** — задачи и доставки автоматически сдвигаются мимо
   пяти молитв и пятничного джума; система объясняет, почему перенесла.
3. **Халяль-статус товара** — статус (халяль/харам/сомнительно) в карточке; на товар
   со статусом «харам» договор оформить нельзя.

Плюс **собственный публичный API** (дословно пример расширяемости из кейса) — почти
гарантированные +3 балла.

## 6. Функциональные возможности (по модулям)

### Financing (ядро) — договоры рассрочки
- Оформление договора мурабаха: себестоимость + фиксированная наценка (суммой или %).
- Автопостроение графика платежей (равные доли, детерминированное округление до копейки).
- **Предпросмотр графика до создания** (расчёт без сохранения) — для мастера договора.
- Регистрация платежей (**любой суммой ≤ остатка**), расчёт остатка, прогресс договора.
- Просрочка **не увеличивает долг**; опциональный фиксированный сбор-садака.
- Досрочное погашение без штрафа. Отмена договора (черновик/активный) — действие владельца.
- Визуальное сравнение «график мурабахи vs обычный кредит с процентом».

### Scheduling — напоминания и доставки
- Календарь с подсветкой окон намаза и джума.
- Создание задач (звонок / доставка / контакт по платежу) с авто-сдвигом мимо намаза.
- Пометка «перенесено из-за намаза» с указанием причины.
- Жизненный цикл задачи: **запланирована → выполнена / отменена**, а просрочка
  показывается как производный статус, если время уже прошло и задача ещё не закрыта.

### Catalog — товары
- Карточка товара: название, категория, закупочная цена, **халяль-статус**.
- Блокировка договора на товар со статусом «харам».

### CRM — клиенты
- Карточка клиента: контакты, документ для договора, список договоров и платежей.

### IAM — доступ
- Пользователи и роли (владелец / менеджер), вход по JWT, пароли через bcrypt.
- Управление API-ключами для публичного API.

### Public API (+3) — интеграции
- `/api/v1/*` с авторизацией по `X-API-Key`.
- Эндпоинты: создать договор (по ссылкам на существующие клиента и товар), получить статус
  платежей по договору.
- OpenAPI + Swagger UI на `/swagger/`.

## 8. Технологический стек

**Бэкенд** (детали — в `ARCHITECTURE.md`):
Go 1.23+, chi, PostgreSQL 16, pgx/v5 + sqlc, goose, shopspring/decimal, slog,
golang-jwt, swaggo/swag, go-prayer (за портом). Модульный монолит, чистая
архитектура, тактический DDD. Внутренний API — `/api/app` (JWT), публичный — `/api/v1`
(`X-API-Key`).

**Фронтенд** (SPA, общается с Go-API по REST; детали — в `web/ARCHITECTURE.md`,
`web/CLAUDE.md`, `web/DESIGN_SYSTEM.md`):
React 18 + TypeScript + Vite, Tailwind + shadcn/ui, React Router, TanStack Query
(серверное состояние), react-hook-form + zod (формы), recharts (графики и сравнение).
**Типы API генерятся из OpenAPI** (`swagger.json` → `openapi-typescript`) — типобезопасный
контракт end-to-end, фронтовый аналог `sqlc`. Архитектура — Feature-Sliced Design,
зеркалит bounded contexts бэкенда.

**Инфраструктура:**
Docker (multi-stage → distroless), `docker-compose.yml` (app + postgres),
Makefile. Запуск проекта — одна команда.

## 9. Архитектура (кратко)

Модульный монолит: один деплоируемый бинарь, внутри — изолированные bounded contexts
(`financing`, `scheduling`, `catalog`, `crm`, `iam`) с границами через интерфейсы.
Зависимости направлены внутрь: `http → app → domain ← infra`. Любой модуль вынимается
в микросервис без переписывания домена. Две поверхности HTTP: внутренний API приложения
(`/api/app`, JWT) и публичный API (`/api/v1`, `X-API-Key`). Полное описание — в
`ARCHITECTURE.md`, доменная модель — в `docs/DOMAIN.md`.

## 10. Ключевые доменные правила

- **Запрет рибы:** сумма обязательства фиксируется при создании и не зависит от времени.
- **Садака вместо пени:** сбор за просрочку фиксированный, идёт на благотворительность,
  не продавцу, и не меняет график/остаток. Начисляет владелец.
- **Намаз:** задачи не попадают в окна молитв и джума (мазхаб шафиитский, Грозный).
- **Халяль:** договор только на халяль/сомнительный товар, не на «харам».
- **Деньги:** в домене — `decimal`, во внутренних расчётах — копейки `int64`, в БД —
  `NUMERIC(18,2)`, на границе API — строка-decimal. Никакого `float`.

Точные формулы, инварианты и тест-кейсы — в `.claude/skills/murabaha-engine/SKILL.md`
и `.claude/skills/namaz-scheduler/SKILL.md`.

## 11. Нефункциональные требования

- **Развёртывание:** `docker compose up`, миграции прогоняются на старте.
- **Безопасность:** JWT для приложения (`/api/app`), API-ключи для публичного API
  (`/api/v1`), bcrypt для паролей, секреты только из окружения.
- **Корректность денег:** `NUMERIC(18,2)` в БД, `decimal`/копейки в коде, строка-decimal в
  API, инвариант «копейки сходятся».
- **Наблюдаемость:** структурные логи `slog`.
- **Тесты:** table-driven на всю доменную математику.

## 12. Публичный API (+3)

- База: `/api/v1/`, заголовок `X-API-Key` (отдельно от внутреннего `/api/app` на JWT).
- `POST /api/v1/contracts` — создать договор рассрочки. Принимает **ссылки на существующие**
  `client_id` и `product_id` + условия (`cost_price`, `markup`, `down_payment`,
  `installments`, `cadence`, `start_date`). Клиент и товар создаются заранее (через
  приложение либо сидируются для демо).
- `GET /api/v1/contracts/{id}/payments` — статус и график платежей.
- Документация: Swagger UI `/swagger/`.
- На демо: создаём договор «от имени внешнего маркетплейса» вживую через Swagger
  (с тестовым `X-API-Key`, по заранее заведённым клиенту и товару).

## 13. Сценарий демо (3 минуты)

1. **Хук:** ниша и проблема рибы (10 сек).
2. Оформляем договор в мастере → предпросмотр графика → «цена фиксирована, 0% риба».
3. Открываем календарь → создаём доставку → система сдвигает её мимо Магриба и поясняет.
4. Показываем реестр садаки (штраф ушёл на благотворительность).
5. Открываем Swagger → создаём договор «извне» через `/api/v1` (X-API-Key) → расширяемость доказана.
6. Слайд архитектуры → закрытие: «логика, которой нет ни у одной CRM, в нише, в которую
   регион вкладывает миллиарды».

## 14. Дорожная карта после MVP

- Аналитика портфеля договоров и просрочек.
- Мобильное приложение менеджера.
- Вынос `financing` в отдельный сервис при росте нагрузки.
